# Конспект: Spring MVC CRUD With Spring Security

## Главная идея

Это учебный проект, где обычное CRUD-приложение на Spring Boot расширено авторизацией и разграничением прав через Spring Security.

Основная концепция обучения:

- как устроен MVC-поток в Spring Boot;
- как связать `Controller -> Service -> Repository -> DB`;
- как подключить `Spring Security` к данным из базы;
- как ограничить доступ к страницам по ролям;
- как совместить CRUD пользователей и login/logout в одном приложении.

## Что реализует проект

- Администратор работает со списком пользователей на `/admin`.
- Обычный пользователь видит свою страницу на `/user`.
- Доступ к URL контролируется ролями `ROLE_ADMIN` и `ROLE_USER`.
- Пользователи и роли хранятся в MySQL.
- Шаблоны интерфейса сделаны на Thymeleaf.
- После логина пользователь попадает на нужную страницу через `LoginSuccessHandler`.

## Структура по слоям

### 1. `model`

Содержит сущности:

- `User` - пользователь приложения;
- `Role` - роль пользователя.

В этом проекте:

- `User` реализует `UserDetails`;
- `Role` реализует `GrantedAuthority`.

Это нужно, чтобы Spring Security мог напрямую использовать объекты приложения для аутентификации и авторизации.

### 2. `repository`

Слой доступа к данным:

- `UserRepository`
- `RoleRepository`

Здесь используется `Spring Data JPA`.

Repository отвечает за:

- поиск пользователя по логину;
- получение списка пользователей;
- создание, обновление и удаление пользователя;
- получение ролей.

### 3. `service`

Сервисный слой:

- `UserService`
- `UserServiceImpl`

Его задачи:

- работать в транзакциях через `@Transactional`;
- шифровать пароль при создании и обновлении;
- загружать пользователя для Spring Security через `loadUserByUsername(...)`;
- координировать CRUD-операции.

Фактически `UserServiceImpl` работает как бизнес-слой и как `UserDetailsService`.

### 4. `controller`

Главный web-слой:

- `UserController`

Он обрабатывает маршруты:

- `GET /` -> редирект на `/user`;
- `GET /admin` -> список пользователей и форма добавления;
- `POST /admin` -> создание пользователя;
- `POST /admin/{id}/delete` -> удаление;
- `GET /admin/{id}/edit` -> форма редактирования;
- `POST /admin/{id}` -> обновление пользователя;
- `GET /user` -> страница текущего пользователя.

## Как здесь работает Spring MVC

Схема запроса:

```text
Браузер
  -> Security Filter Chain
  -> Controller
  -> Service
  -> Repository
  -> Spring Data JPA / Hibernate
  -> MySQL
```

Дальше ответ возвращается обратно, а Thymeleaf строит HTML-страницу.

Ключевая мысль:

- `Controller` принимает HTTP-запрос;
- `Service` содержит прикладную логику;
- `Repository` работает с базой;
- `ModelMap` передает данные в шаблон;
- Thymeleaf рендерит страницу.

## Как здесь работает Spring Security

Безопасность настраивается в `WebSecurityConfig`.

Основные правила:

- `/admin/**` доступен только для `ADMIN`;
- `/user` доступен для `USER` и `ADMIN`;
- остальные запросы требуют аутентификации.

### Аутентификация

Приложение не использует `inMemoryAuthentication`.

Вместо этого:

- Spring Security берет пользователя из базы;
- `UserService` ищет его по username;
- пароль проверяется через `BCryptPasswordEncoder`;
- провайдер аутентификации настроен через `DaoAuthenticationProvider`.

### Авторизация

После входа система смотрит на роли пользователя:

- если есть `ROLE_ADMIN`, можно открыть `/admin`;
- если есть `ROLE_USER`, можно открыть `/user`.

### Успешный логин

`LoginSuccessHandler` отправляет пользователя на правильную страницу после входа:

- админ -> `/admin`;
- обычный пользователь -> `/user`.

## Как устроен CRUD в этом проекте

### Create

Админ отправляет форму на `POST /admin`.

Что делает контроллер:

- получает объект `User` через `@ModelAttribute`;
- получает список `roleIds`;
- загружает роли;
- передает пользователя в `service`;
- делает `redirect:/admin`.

Что делает сервис:

- шифрует пароль;
- сохраняет пользователя.

### Read

`GET /admin` загружает:

- список всех пользователей;
- пустой объект для формы;
- список ролей.

`GET /user` показывает текущего авторизованного пользователя через `@AuthenticationPrincipal`.

### Update

`POST /admin/{id}`:

- получает измененные данные из формы;
- передает пользователя в `service`;
- сервис либо сохраняет старый пароль, либо шифрует новый;
- сохраняет изменения.

### Delete

`POST /admin/{id}/delete` удаляет пользователя по `id`.

## Как теперь работает инициализация БД

Раньше начальные роли и пользователи создавались Java-классом `DataInitializer`.

Теперь это делает Liquibase:

- changelog-файлы лежат в `src/main/resources/db/changelog`;
- Liquibase сам создаёт таблицы;
- Liquibase сам вставляет стартовые роли и пользователей;
- при старте он записывает, какие миграции уже были применены.

За счет этого:

- схема базы хранится в проекте явно;
- изменения БД становятся версионируемыми;
- одинаковая схема поднимается у всех разработчиков;
- не нужно держать инициализацию таблиц в Java-коде.

## Как работает Liquibase поверхностно

Liquibase - это инструмент миграций базы данных.

Идея простая:

1. Ты описываешь изменения схемы в changelog-файлах.
2. Приложение при старте проверяет, какие changeSet уже применялись.
3. Новые changeSet выполняются один раз.
4. Повторно уже выполненные миграции Liquibase не запускает.

Обычно через Liquibase выносят:

- создание таблиц;
- изменение колонок;
- индексы;
- внешние ключи;
- начальные справочники и seed-данные.

## Как работает Docker здесь

Docker нужен, чтобы быстро поднять одинаковую MySQL без ручной установки в систему.

В этом проекте:

- `docker-compose.yml` поднимает контейнер MySQL;
- база доступна на `localhost:3308`;
- данные сохраняются в volume `mysql_data`;
- приложение подключается к контейнеру по обычному JDBC URL.

Базовая команда:

```text
docker compose up -d mysql
```

## Важные аннотации

- `@SpringBootApplication` - точка входа и автоконфигурация Spring Boot.
- `@Controller` - web-обработчик запросов.
- `@Service` - сервисный слой.
- `@Repository` / Spring Data repository - слой доступа к данным.
- `@Entity` - JPA-сущность.
- `@Transactional` - транзакции.
- `@GetMapping` / `@PostMapping` - маршруты.
- `@AuthenticationPrincipal` - текущий авторизованный пользователь.
- `@ModelAttribute` - сбор данных формы в объект.
- `@PathVariable` - чтение параметра из URL.
- `@RequestParam` - получение параметров запроса.

## Что важно понять по этому проекту

1. Spring Boot упрощает запуск и конфигурацию, но архитектура MVC остается той же.
2. Spring Security можно подключить к своей таблице пользователей, если `User` и `Role` согласованы с его интерфейсами.
3. Роли управляют доступом к URL, а не просто отображаются в базе.
4. CRUD-логика и security-логика должны работать вместе: при создании пользователя нужно назначать роли, а пароль должен шифроваться в сервисе.
5. Liquibase отвечает за жизненный цикл схемы БД, а Hibernate в этом проекте только валидирует соответствие сущностей.

## Короткое резюме

Главная тема этого обучения: не просто CRUD, а построение полноценного `Spring MVC CRUD` приложения с `Spring Security`, ролями, MySQL, миграциями через Liquibase и запуском базы в Docker.
